TDD（テスト駆動開発）ワークフローを実行します。

**対象**: $ARGUMENTS

`dev-tester` エージェントを使って、以下の TDD サイクルを厳密に実施してください。

---

## TDD サイクル

### フェーズ 1: インターフェース定義（SCAFFOLD）

まずソースコードと `ARCHITECTURE.md` を読んで、テスト対象の関数・モジュールを特定してください。
型・関数シグネチャを定義します（実装は空 or `raise NotImplementedError`）。

### フェーズ 2: 失敗するテストを書く（RED）

テストファイルを作成し、**実装より先にテストを書いてください**。

以下のテストケースを必ず網羅する：
- ハッピーパス（正常系）
- エッジケース（空・null・最大値・最小値）
- エラーケース（不正入力・例外）

テストを実行して **失敗することを確認** してください：
```bash
# Python
pytest tests/ -v

# JavaScript/TypeScript
npm test
```

✅ テストが失敗した → 次のフェーズへ
❌ テストが通った → テストが実装に依存していないか確認する

### フェーズ 3: テストが通る最小限の実装（GREEN）

テストをパスさせる**最小限の**コードを実装してください。
過剰な実装はしない。テストが通れば十分。

テストを再実行して **全テストがパスすることを確認** してください。

### フェーズ 4: コードを改善（REFACTOR）

テストがグリーンのまま、以下の観点でコードを改善してください：
- 可読性の向上（変数名・関数名の明確化）
- 重複の除去（DRY原則）
- パフォーマンスの最適化
- マジックナンバーを定数に置き換え

テストを再実行して **まだパスすることを確認** してください。

### フェーズ 5: カバレッジ確認

```bash
# Python
pytest --cov=src tests/ --cov-report=term-missing

# JavaScript/TypeScript
npm test -- --coverage
```

**目標: 80% 以上**
- 未達成の場合、カバレッジが低い箇所にテストを追加して再実行する

---

## 完了報告

TDD セッション完了後、以下を報告してください：

```
## TDD セッション完了

### サイクル確認
- [x] RED: 失敗するテストを確認
- [x] GREEN: 全テストがパス
- [x] REFACTOR: コード改善完了

### テスト結果
- 成功: N / N
- カバレッジ: XX% (目標 80%)

### 作成したテストファイル
- tests/...

### 次のステップ
次の機能・シナリオがあれば RED → GREEN → REFACTOR を繰り返してください。
```
